# Life Hub Nexus Relationship Specification

Version: 1.0 (T-013 baseline)
Date: 2026-07-19

## 1. Purpose

This document replaces the ad-hoc `linked*Ids` arrays with one explicit, feature-neutral relationship contract. It defines what a relationship is, how it is owned, how it may be queried, and how it behaves when either endpoint is deleted. The contract is the input to T-014, which will isolate the existing cleanup logic behind a tested boundary.

The specification follows DDD strategic and tactical patterns:

- **Relationships are a bounded context** that protects its own model from feature-specific entity types.
- **Anti-corruption layer** between feature entities and generic link records: feature contexts see only their own `linked*Ids` arrays today; the relationship repository sees only source type, source id, target type, target id, relation type, and owner.
- **Published language** is a small set of relationship primitives (`Relationship`, `EntityType`, `RelationType`, `RelationshipRepository`) consumed by all feature contexts.

## 2. Relationship model

A relationship is a directed association between two entities. It is not an aggregate root of either endpoint; it is a link record with its own identity and lifecycle.

### 2.1 Core attributes

| Attribute | Type | Description |
|-----------|------|-------------|
| `id` | `string` | Unique relationship identifier, generated by the same `IdGenerator` port used by other bounded contexts. |
| `sourceType` | `EntityType` | The context/entity that owns the outbound side of the link (e.g. `task`). |
| `sourceId` | `string` | Identifier of the source entity. |
| `targetType` | `EntityType` | The context/entity that is the inbound target of the link. |
| `targetId` | `string` | Identifier of the target entity. |
| `relationType` | `RelationType` | Semantic label of the link (default `related`). |
| `ownerId` | `string` | Actor id from the Identity context. Used for authorization and scoping. |
| `createdAt` | `string` | ISO timestamp from a `Clock` port. |
| `deletedAt` | `string \| undefined` | Soft-delete timestamp for sync tombstones. |

### 2.2 Entity types

```typescript
type EntityType =
  | 'task'
  | 'event'
  | 'note'
  | 'person'
  | 'transaction'
  | 'folder'
  | 'budget';
```

Notes:

- `folder` and `budget` are included for future completeness but are not yet linked by any feature code.
- A relationship is allowed between any two distinct entity types and also between two entities of the same type when business rules justify it (e.g. a task and its parent task in the future).

### 2.3 Relation types

```typescript
type RelationType =
  | 'related'         // generic association
  | 'scheduled-for'   // event ↔ task
  | 'mentioned-in'    // note ↔ task/person/event
  | 'paid-for'          // transaction ↔ task/person
  | 'attends'           // person ↔ event
  | 'created-from';     // note created from a task or event
```

Notes:

- The default relation is `related`.
- Relation types are semantic hints for UI grouping; they do not change deletion behavior.
- New relation types are added by extending the union and updating the `RelationshipPolicy` whitelist.

- All current links are **optional associations**, not structural ownership.

- Each `linked*Ids` array is a bidirectional projection of the same underlying relationship record.

## 4. Deletion semantics

### 4.1 Association vs structural ownership

| Concept | Definition | Example | Deletion rule |
|---------|------------|---------|---------------|
| **Association** | A non-owning link between two independent aggregates. | A task linked to a note. | Removing the link or deleting either endpoint does not delete the other endpoint. |
| **Structural ownership** | One aggregate is responsible for the lifecycle of its child entities. | A task owns its subtasks; a person owns its interactions. | Deleting the parent removes the children. |

### 4.2 Endpoint deletion

When an entity is deleted:

1. The entity's own aggregate deletes it (structural children may be deleted with it).
2. The Relationships context removes every relationship where the deleted entity is `sourceId` or `targetId`.
3. The cleanup is **eventual consistency** because feature contexts today update their own `linked*Ids` arrays independently.
4. No other independent entity is deleted as a side effect of removing a relationship.

### 4.3 Current behavior vs target behavior

Current behavior (verified by `artifacts/mobile/__tests__/entity-reference-policy.test.ts`):

- `EntityReferencePolicy` removes the deleted entity id from all inbound `linked*Ids` arrays.
- `ReferenceCleanupService` reads from `AsyncStorage`, applies the policy, and writes updated collections back.
- `ReferenceCleanupOrchestrator` exists but is not imported by any production code.

Target behavior after T-014/T-016:

- `RelationshipPolicy` removes relationships where the deleted entity is source or target.
- Feature contexts project those relationships into their own `linked*Ids` arrays on read.
- Cleanup returns structured results (success / recoverable failure) instead of swallowing errors.

## 5. Query semantics

### 5.1 Reads

- `listByEntity(entityType, entityId)` returns all relationships where `sourceType === entityType && sourceId === entityId` **or** `targetType === entityType && targetId === entityId`.
- The result set is ordered by `createdAt` ascending.
- The default direction is **both** source and target. Consumers may filter direction in application code.

### 5.2 Authorization

- Every relationship is scoped to an `ownerId` derived from the authenticated actor.
- `create`, `remove`, and `listByEntity` must filter by `ownerId`.
- A relationship may never expose data from another owner.
- On the server, the `ownerId` comes from `AuthenticatedActor`; the client does not supply it.

## 6. Proposed public contract

The following interfaces are proposed for implementation in T-014 and beyond. They are **not** production exports until T-014.

```typescript
interface Relationship {
  id: string;
  sourceType: EntityType;
  sourceId: string;
  targetType: EntityType;
  targetId: string;
  relationType: RelationType;
  ownerId: string;
  createdAt: string;
  deletedAt?: string;
}

interface CreateRelationshipInput {
  sourceType: EntityType;
  sourceId: string;
  targetType: EntityType;
  targetId: string;
  relationType?: RelationType;
}

interface RelationshipRepository {
  /**
   * Creates a relationship for the given owner.
   * Idempotent: creating the same source/target/type triple returns the existing record.
   */
  create(
    input: CreateRelationshipInput,
    ownerId: string
  ): Promise<Relationship>;

  /** Soft-removes a single relationship by id. */
  remove(relationshipId: string, ownerId: string): Promise<void>;

  /** Lists all relationships involving the entity, regardless of direction. */
  listByEntity(
    entityType: EntityType,
    entityId: string,
    ownerId: string
  ): Promise<Relationship[]>;

  /** Removes every relationship where the entity is source or target. */
  removeForEntity(
    entityType: EntityType,
    entityId: string,
    ownerId: string
  ): Promise<void>;
}

interface RelationshipPolicy {
  /** Returns true if a link between source and target types is allowed for the relation. */
  canLink(
    sourceType: EntityType,
    targetType: EntityType,
    relationType: RelationType
  ): boolean;

  /** Removes relationships where the deleted entity is the source. */
  onSourceDeleted(
    sourceType: EntityType,
    sourceId: string,
    ownerId: string
  ): Promise<void>;

  /** Removes relationships where the deleted entity is the target. */
  onTargetDeleted(
    targetType: EntityType,
    targetId: string,
    ownerId: string
  ): Promise<void>;
}
```

### 6.1 Transactional vs eventual operations

| Operation | Boundary | Semantics |
|-----------|----------|-----------|
| `create` | Single aggregate (Relationship) | Transactional within the relationship store. |
| `remove` | Single aggregate | Transactional within the relationship store. |
| `removeForEntity` | Relationship store + eventual projection | The relationship records are removed transactionally; feature `linked*Ids` arrays are updated eventually because each context writes its own collection. |
| `listByEntity` | Read-only | No transactional guarantees against concurrent mutations. |

## 7. Storage representation

### 7.1 Local-first (current target)

- An `AsyncStorage` adapter stores a single JSON array under `@lifeos/relationships`.
- The adapter validates each stored shape and returns typed `Result` outcomes.
- Feature contexts keep their own `linked*Ids` arrays until a future migration stores relationships centrally.

### 7.2 Future server/SQLite representation

Two options are documented; the decision is deferred to the schema task that creates the relationships table.

| Option | Schema | Pros | Cons |
|--------|--------|------|------|
| **A. Generic relationship table** | `relationships(source_type text, source_id text, target_type text, target_id text, relation_type text, owner_id text, created_at timestamp, deleted_at timestamp)` | Single table, simple queries, easy to add new entity types. | PostgreSQL cannot enforce foreign keys to multiple entity tables; referential integrity must be application-level or eventual. |
| **B. Per-pair relationship tables** | e.g. `task_event_rel`, `task_note_rel`, ... with proper FKs to each parent table. | Full referential integrity enforced by the database. | More tables, more migrations, no generic `listByEntity` without a union view. |

Important: the task explicitly avoids claiming PostgreSQL can enforce polymorphic foreign keys. Option A requires application-level validation and eventual cleanup; Option B is the integrity-strong alternative. The choice is deferred.

## 8. Anti-corruption layer

The relationship boundary protects feature contexts from generic link details and protects the relationship model from feature-specific types.

```text
Planning Task   ----->   Relationship DTO   <-----   RelationshipRepository
Calendar Event  ----->   (source/target)    <-----   AsyncStorage adapter
Note            ----->                        <-----   (future SQLite/server)
Person          ----->
Transaction     ----->
```

Rules:

- Feature contexts may only pass primitive ids and type labels into the repository.
- The repository never imports React, Expo, or feature context types.
- The repository returns domain-safe results, not raw storage errors.

## 9. Mismatch between specified and current behavior

The following discrepancies were found during T-013 and must be addressed in T-014:

1. **`ReferenceCleanupService` loads notes from the budget storage key.** The code calls `safeParse<{ notes: Note[]; folders: any[] }>(budgetRaw, ...)` where `budgetRaw` is the transactions array. Because arrays have no `notes` property, the expression falls back to `notesRaw` only when `budgetRaw` is an array. When `budgetRaw` is `null` (first run), `budget.notes` is an empty array (truthy), so the actual notes store is ignored. This is a data-loss risk and violates the planned Local Device Cache boundary.
2. **`ReferenceCleanupService` and `EntityReferencePolicy` import feature entity types from React context files.** This violates the mobile layering rule that domain code must not import React contexts.
3. **`ReferenceCleanupOrchestrator` is not imported anywhere.** It appears to be dead code and should either be removed or aligned with the new repository contract in T-014.
4. **`ReferenceCleanupService` swallows parsing errors by returning default values.** Malformed stored data should be reported as a recoverable failure, not silently replaced with an empty default.
5. **Cleanup writes are not atomic.** A crash between writes may leave some `linked*Ids` arrays updated and others stale.
